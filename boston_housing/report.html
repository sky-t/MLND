Implementation: Define a Performance Metric¶

It is difficult to measure the quality of a given model without quantifying its performance over training and testing. This is typically done using some type of performance metric, whether it is through calculating some type of error, the goodness of fit, or some other useful measurement. For this project, you will be calculating the coefficient of determination, R2, to quantify your model's performance. The coefficient of determination for a model is a useful statistic in regression analysis, as it often describes how "good" that model is at making predictions.

The values for R2 range from 0 to 1, which captures the percentage of squared correlation between the predicted and actual values of the target variable. A model with an R2 of 0 is no better than a model that always predicts the mean of the target variable, whereas a model with an R2 of 1 perfectly predicts the target variable. Any value between 0 and 1 indicates what percentage of the target variable, using this model, can be explained by the features. A model can be given a negative R2 as well, which indicates that the model is arbitrarily worse than one that always predicts the mean of the target variable.

Question 8 - Cross-Validation¶

  • What is the k-fold cross-validation training technique?

  • What benefit does this technique provide for grid search when optimizing a model?

Hint: When explaining the k-fold cross validation technique, be sure to touch upon what 'k' is, how the dataset is split into different parts for training and testing and the number of times it is run based on the 'k' value.

When thinking about how k-fold cross validation helps grid search, think about the main drawbacks of grid search which are hinged upon using a particular subset of data for training or testing and how k-fold cv could help alleviate that. You can refer to the docs for your answer.

Answer: The k-fold cross validation is a technique to help avoid model overfitting and aid in model generalization. It works by taking the body of data allocated for training and validation, then splitting it into "k" number of equal sized groups, known in this context as "folds". Imagine a piece of paper representing the data you have reserved for training and validation, and the folding it into equal sized folds. It then proceeds like this:

  • The algorithm then takes the first fold, sets it aside for validation, and then combines the remainder of the data for the other folds into a training set.
  • The model is trained on that training set, and scored (for error rate) on the validation set (the fold set aside).
  • Then, the next fold is set aside for validation, and the remaining folds are combined into a training set.
  • The model is again trained on that training set, and scored on the new validation set.
  • This process is repeated until all folds have been had a turn at being the validation set.
  • The error scores from each of training-score iterations are averaged to come with a single score called the cross validation error

This cross validation techqnique is combined with the grid search technique in GridSearchCV. Each combination of test hyperparameters (spot on the grid) is scored with the k-fold approach to training-validation described above, and then the optimal model with best hyperparameters can be identified from the lowest cross validation error.

One benefit of the k-fold CV technique is that the modeler can perform validation without having to lose a significant amount of data that would have to be held out strictly for valiation. Since the algorithm randomly splits the training data into a training and validation set k fold times, you don't have to set aside the validation set aside from training. This also helps guard against overfitting since as your dataset gets smaller, you risk training the model to "memorize" on a smaller set of data, and not be as generalizable. The process of using k folds of training/validation splits is that the model space has seen some diversity of data and the best parameters are picked out, which should end up the most generalizable.

Contrast this with a more traditional approach to training, validation, and testing where you set aside data for all 3 purposes, and do not take a cross validation approach. In this scenario, there is more risk that, due to chance, you might get unlucky and have skewed data in the training set. One can imagine a scenario like fraud detection where the class imbalance between "good" transactions and "fraudulent" transactions, is large, with very few fraudulent transactions. In the non-cross validation approach, those few fraudulent data points could end up primarily in the testing or validation sets, giving the model very few valid points to train on. Cross validation helps guard against (but not eliminate) this possibility by exposing the models to different validation sets that get exposed to all of the training data. This aids in increasing the chance that the model selected as best, is truly the most generalizable among the choices explored.

from sklearn.tree import DecisionTreeRegressor
from sklearn.metrics import make_scorer
from sklearn.grid_search import GridSearchCV


def fit_model(X, y):
    """ Performs grid search over the 'max_depth' parameter for a 
        decision tree regressor trained on the input data [X, y]. """
    
    # Create cross-validation sets from the training data
    # sklearn version 0.18: ShuffleSplit(n_splits=10, test_size=0.1, train_size=None, random_state=None)
    # sklearn versiin 0.17: ShuffleSplit(n, n_iter=10, test_size=0.1, train_size=None, random_state=None)
    cv_sets = ShuffleSplit(X.shape[0], n_iter = 10, test_size = 0.20, random_state = 0)

    # TODO: Create a decision tree regressor object
    regressor = DecisionTreeRegressor()

    # TODO: Create a dictionary for the parameter 'max_depth' with a range from 1 to 10
    params = {'max_depth':range(1,11)}

    # TODO: Transform 'performance_metric' into a scoring function using 'make_scorer' 
    scoring_fnc = make_scorer(performance_metric)

    # TODO: Create the grid search cv object --> GridSearchCV()
    # Make sure to include the right parameters in the object:
    # (estimator, param_grid, scoring, cv) which have values 'regressor', 'params', 'scoring_fnc', and 'cv_sets' respectively.
    grid = GridSearchCV(estimator=regressor,
                       param_grid=params,
                       scoring=scoring_fnc,
                       cv=cv_sets)

    # Fit the grid search object to the data to compute the optimal model
    grid = grid.fit(X, y)

    # Return the optimal model after fitting the data
    return grid.best_estimator_

Sensitivity¶

An optimal model is not necessarily a robust model. Sometimes, a model is either too complex or too simple to sufficiently generalize to new data. Sometimes, a model could use a learning algorithm that is not appropriate for the structure of the data given. Other times, the data itself could be too noisy or contain too few samples to allow a model to adequately capture the target variable — i.e., the model is underfitted.

Question 11 - Applicability¶

  • In a few sentences, discuss whether the constructed model should or should not be used in a real-world setting.

Hint: Take a look at the range in prices as calculated in the code snippet above. Some questions to answering:

  • How relevant today is data that was collected from 1978? How important is inflation?
  • Are the features present in the data sufficient to describe a home? Do you think factors like quality of apppliances in the home, square feet of the plot area, presence of pool or not etc should factor in?
  • Is the model robust enough to make consistent predictions?
  • Would data collected in an urban city like Boston be applicable in a rural city?
  • Is it fair to judge the price of an individual home based on the characteristics of the entire neighborhood?

Answer: I would not use this model for a variety of reasons summarized below:

  • Data collected from 1978 may not be so relevant today. Prices are a function of buyer psychology and the actual products themselves. For one thing, the composition of housing stock has changed since 1978 and so the data is not representative of today's housing. In addition, today's buyer may not be yesterday's buyer, as demographics accross a broad spectrum of socioeconomic criteria have shifted. In addition, an inflation factor has been applied to the 1978 prices. This approach then makes the assumption that the same inflation has occurred uniformly amoung all housing in the dataset. It's always possible that inflation has occurred non-uniformly for all types of housing in the dataset, further invalidating the model.

  • The dataset only has 3 features: # of rooms, neighborhood poverty level, and class ratio. From personal experience, square footage, number of bathrooms, size of lot, and internal finish quality are also important factors that could be explored and potentially added to the model.

  • The sensitivity study shows that the model varies up to ~ $70K when trained/tested on different data. That's quite a large number and more work should be done to further desensitize the model if possible. That large of a swing in prediction value could cause a client to lose a house or overbid.

  • The model likely would not be generalizable to rural areas since the dataset was exclusively sampled from urban data -- it's simply not representative and would not make accurate predictions.

  • I do feel that it's fair to somewhat judge a house by its neighborhood, since those neighborhood features do have an impact on price. The key thing to remember is that those factors are not the only ones, and as long as we understand them in context, they are useful and can be used for estimating price.
